### Added

- Stage 6: `bristol_ml.evaluation.plots` — seven diagnostic-plot helpers for the evaluation layer. Four residual diagnostics (`residuals_vs_time`, `predicted_vs_actual`, `acf_residuals`, `error_heatmap_hour_weekday`), a forecast overlay (`forecast_overlay`), an empirical-quantile uncertainty-band overlay (`forecast_overlay_with_band`), and the fixed-window NESO retrospective bar chart (`benchmark_holdout_bar`) that consumes `NesoBenchmarkConfig.holdout_start/_end` (resolving the Stage 4 carry-over). All helpers are model-agnostic (inputs are `pd.Series` / `pd.DataFrame`, never a `Model` object), accept an optional `ax=` for composability, and inherit their default figsize (12×8) from `PlotsConfig`. Okabe-Ito qualitative palette (Wong 2011 *Nature Methods*), `cividis` sequential colormap, and `RdBu_r` diverging colormap are injected into `plt.rcParams` at module import; `tab10` (matplotlib's default) is explicitly rejected as not CVD-safe. Module CLI `python -m bristol_ml.evaluation.plots --help` prints the exported helper surface, the active palette, and the rcParams written at import time.
- Stage 6: `bristol_ml.evaluation.harness.evaluate(..., return_predictions=True)` — backward-compatible one-off tolerance (plan D9) that returns a `(metrics_df, predictions_df)` tuple with one row per (fold, forecast-horizon-index) in `predictions_df`. Columns pinned: `["fold_index", "test_start", "test_end", "horizon_h", "y_true", "y_pred", "error"]`. Stage 4 call sites are unaffected (default `return_predictions=False` keeps the single-DataFrame return). `@overload` stubs narrow the return type under `Literal[True]` / `Literal[False]`. An **architectural-debt note** in `src/bristol_ml/evaluation/CLAUDE.md` records this as a single-flag concession — the second boolean extension to `evaluate()` triggers a promotion to a first-class `EvaluationResult` dataclass or a typed `evaluate_v2`.
- Stage 6: `matplotlib>=3.8,<4` and `seaborn>=0.13,<1` promoted from `[dependency-groups].dev` to `[project].dependencies` in `pyproject.toml` so `python -m bristol_ml.evaluation.plots --help` works on a non-dev install (plan D3). `uv.lock` regenerated.
- Stage 6: Hydra config group `conf/evaluation/plots.yaml` with user-tweakable defaults (`figsize: [12.0, 8.0]`, `dpi: 110`, `display_tz: "Europe/London"`, `acf_default_lags: 168`). Pydantic `PlotsConfig` in `conf/_schemas.py`; `EvaluationGroup.plots: PlotsConfig` non-optional. Helper-function defaults match config defaults so construction without Hydra still works.
- Stage 6: `notebooks/04_linear_baseline.ipynb` — six Stage 6 appendix cells (indices 13–18) appended after the Stage 4 shipping view. Cells 0–12 (including the inline-matplotlib Cell 11) are byte-identical to Stage 4 per the 2026-04-20 D11 human mandate; the appendix demonstrates the four hero diagnostics in a 2×2 grid, the `forecast_overlay` helper, the uncertainty-band overlay exercising `evaluate(return_predictions=True)`, and the `benchmark_holdout_bar` (guarded on a warm NESO cache). All appendix cells carry a `stage6-appendix` metadata tag so the scratch append is idempotent. Runs end-to-end via `jupyter nbconvert --execute` inside the 180 s budget.
- Stage 6: spec-derived test suite — 41 `test_plots.py` tests (palette-injection, `ax=` composability, `_require_tz_aware_datetime_index` guard, ACF marker placement, DST spring-forward / fall-back gates, `forecast_overlay_with_band` quantile arithmetic, `benchmark_holdout_bar` lazy-import monkeypatch path, model-agnostic signature contract) and 6 `test_harness.py` additions pinning the D9 predictions emission (column order, dtypes, `horizon_h` zero-based per fold, `error = y_true - y_pred` identity). ### Fixed

- Stage 6 (Phase 3 review response): `bristol_ml.evaluation.plots.forecast_overlay` silently drew the second prediction line in the same colour as the Actual series. The palette slice `OKABE_ITO[1:] + OKABE_ITO[3:]` included `OKABE_ITO[2]` (sky blue, the Actual colour) at position 1; corrected to `OKABE_ITO[1:2] + OKABE_ITO[3:]`. Notebook Cell 15 (which overlays Naive / Linear / NESO) now renders all four series in distinct colours. Regression test added. (Phase 3 review B1.)
- Stage 6 (Phase 3 review response): `bristol_ml.evaluation.plots.forecast_overlay_with_band` silently drew an inverted band when the caller passed a reversed `quantiles=(high, low)` tuple. Positional `iloc[:, 0]` / `iloc[:, 1]` access on the unstacked groupby-quantile took whichever column order pandas preserved from the input list, producing a negative-height `fill_between`. Now rejects any `quantiles` that is not a strictly ordered `(lo, hi)` pair within `[0.0, 1.0]` with a `ValueError`; label-based column access added as defence-in-depth. Regression tests added. (Phase 3 review N1.)

- Stage 4: `bristol_ml.models` — new models layer. - Stage 4: `bristol_ml.evaluation.metrics` — four pure `(y_true, y_pred) -> float` point-forecast metrics (`mae`, `mape`, `rmse`, `wape`) as fractions (DESIGN §5.3), plus a `METRIC_REGISTRY` mapping lowercase name → callable. MAPE raises on any zero in `y_true` (D8); WAPE uses `Σ|y_true − y_pred| / Σ|y_true|` (Kolassa & Schütz 2007 / Hyndman) and raises on `Σ|y_true| == 0` (D9). Array-likes (`np.ndarray`, `pd.Series`, `list[float]`) are accepted and coerced at a shared `_coerce_and_validate` gate that rejects length mismatches and NaN. Module CLI `python -m bristol_ml.evaluation.metrics --help`.
- Stage 4: `bristol_ml.evaluation.harness` — `evaluate(model, df, splitter_cfg, metrics, *, target_column="nd_mw", feature_columns=None) -> pd.DataFrame` fold-level evaluator that iterates `rolling_origin_split_from_config`, calls `model.fit` / `model.predict` per fold, and returns a long-form per-fold DataFrame (`fold_index`, `train_end`, `test_start`, `test_end`, plus one float column per metric). Defaults `feature_columns` to every weather column in `assembler.WEATHER_VARIABLE_COLUMNS` so the harness and `LinearModel` cannot disagree on "what are the features?". One structured loguru INFO line per fold plus a completion summary with total folds, elapsed time, and per-metric mean ± std.
- Stage 4: `bristol_ml.evaluation.benchmarks` — `align_half_hourly_to_hourly(df, *, aggregation="mean", value_columns=...)` D4 aggregator (HH → hourly, `mean` default, `first` alternative) and `compare_on_holdout(models, df, neso_forecast, splitter_cfg, metrics, *, aggregation="mean", target_column="nd_mw", feature_columns=None) -> pd.DataFrame` three-way benchmark helper. Runs each candidate through `harness.evaluate` and scores the aligned NESO series against its own outturn on the intersection of fold test-periods ∩ forecast coverage; returns a DataFrame indexed `[*sorted(models), "neso"]` with one column per metric. Empty intersections raise rather than returning an empty frame. Module CLI `python -m bristol_ml.evaluation.benchmarks --help`.
