# Stage 12 — Minimal serving endpoint

**Status:** Intent (immutable once stage is shipped)
**Depends on:** Stage 9
**Enables:** Stage 19 (orchestration)

## Purpose

Add a tiny HTTP service that loads a model from the registry and answers prediction requests. The point is not production serving — it is to demonstrate, concretely, that the artefacts produced by the training pipeline can cross a process boundary and meet a request. This is the CD4ML "training and serving are different pipelines" lesson made tangible.

## Scope

In scope:
- A minimal HTTP application with a single prediction endpoint that loads a named model from the Stage 9 registry and returns a forecast for a given set of inputs.
- Input and output schemas for the prediction endpoint, defined typed and validated at the boundary.
- A small notebook or curl example showing the service answering a request end-to-end.
- Documentation of how to start the service locally and how to ask it questions.

Out of scope:
- Authentication, authorisation, rate limiting.
- Multi-user concurrent request handling at scale.
- Deployment anywhere other than localhost.
- A UI beyond whatever interactive docs the HTTP framework produces.
- Batch prediction endpoints (the registry already supports this pattern; serving is about demonstrating the boundary, not replicating training).
- Model hot-reload.

## Demo moment

A facilitator starts the service, curls it with a sample payload, and gets a forecast back — all within a minute. The gap between "training a model" and "running a model somewhere that isn't the training notebook" becomes small and concrete.

## Acceptance criteria

1. The service starts on a clean machine without configuration beyond pointing at a registry location.
2. A request with valid inputs returns a prediction; a request with invalid inputs returns a clear error.
3. The same model served by the service produces predictions identical to those produced by loading and running the model directly.
4. Input/output schemas are machine-readable (the HTTP framework's schema export is sufficient).
5. A smoke test exercises the endpoint with a small fixture.

## Points for consideration

- Which model to serve by default. The best model from the registry is one answer; a named configuration is another. The choice shapes whether this looks like "serve the best thing we have" or "serve whatever was asked for."
- The request schema needs to supply the features the model expects. Since the feature table is generated by the assembler, there is a real question of whether the service should accept raw inputs and assemble features itself (closer to real-world use), or accept already-assembled feature rows (simpler, but leaks the training assumption). Both are defensible.
- Training-serving skew. If the service and the training pipeline compute features differently, predictions will drift silently. This is the canonical MLOps failure; pointing at it at a meetup is itself educational.
- Latency is not a goal here, but it will come up. A single-request round trip of under a second on a laptop is plausible for all models in the roster; if any model is much slower, that's a talking point.
- Whether to support multiple models behind different endpoints or behind a model-name parameter. The latter is more flexible; the former is easier to demo.
- Logging of requests and predictions. Minimal logging is easy to add and useful for Stage 18's drift monitoring, which needs production predictions to analyse. Too much logging clutters; too little leaves drift work stranded.
- Deployment. Out of scope for this stage, but the structure of the service — pure Python, registry-driven, typed boundaries — should not preclude a future "actually deploy this" stage.

## Dependencies

Upstream: Stage 9 (registry, which is the source of models to serve).

Downstream: Stage 18 (drift monitoring, which benefits from logged predictions), Stage 19 (orchestration, which may schedule batch scoring runs through the same artefacts).

## Out of scope, explicitly deferred

- Authentication, HTTPS, deployment.
- Batch scoring endpoints.
- Model versioning semantics beyond "ask for a model by name."
- A UI.
