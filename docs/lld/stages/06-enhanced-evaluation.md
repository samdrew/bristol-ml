# Stage 6 — Enhanced evaluation & visualisation

## Goal

Land a colourblind-safe diagnostic-plot helper library for the evaluation layer so that every Stage 7+ modelling stage inherits a richer diagnostic surface without reinventing matplotlib; extend the Stage 4 rolling-origin harness with an opt-in per-fold predictions emission so the uncertainty-band helper has a principled data source; wire up the latent `NesoBenchmarkConfig.holdout_start/_end` fields via a fixed-window retrospective bar chart (closing the Stage 4 carry-over); and demonstrate the surface by appending six cells to the Stage 4 notebook — without modifying Stage 4's shipped Cell 11 inline matplotlib (plan D11, 2026-04-20 human mandate). No new models; no new metrics.

## What was built

## Design choices made here

- **D1 — matplotlib + seaborn, nothing else.** No plotly; no altair. matplotlib renders on github.com without a runtime; seaborn is a 295 kB layer for the `colorblind` palette helper and the 24×7 heatmap; every existing notebook already uses matplotlib. External research §R1 pinned this decision.
- **D2 — Okabe-Ito palette injected into `plt.rcParams` at import time.** `OKABE_ITO` is the Wong 2011 *Nature Methods* 8:441 colourblind-safe qualitative palette; `cividis` is the CVD-safe perceptually-uniform sequential colormap; `RdBu_r` is the diverging default that preserves red/blue affect for signed residuals. `tab10` (matplotlib's default) is explicitly rejected — not colourblind-safe. Facilitators who want a bespoke palette call `plt.rcdefaults()` or `plt.rcParams.update(...)` after the import; the helpers honour the active rcParams and never hard-code colour lists inside a helper body.
- **D3 — matplotlib + seaborn promoted to `[project].dependencies`.** AC-10 requires `python -m bristol_ml.evaluation.plots --help` to work on a non-dev install; DESIGN §2.1.1 requires every module run standalone. The alternative (lazy-import-inside-function) was rejected as adding non-locality for 1.4 MB saved. `statsmodels` is already runtime (Stage 2's LOWESS); matplotlib joins it.
- **D4 — module name is `src/bristol_ml/evaluation/plots.py`.** Matches the existing peer filenames (`splitter.py`, `metrics.py`, `harness.py`, `benchmarks.py`).
- **D5 — default figsize `(12.0, 8.0)`, Hydra-configurable via `PlotsConfig.figsize`.** Human mandate 2026-04-20: wider than the matplotlib 6.4×4.8 baseline for meetup-projector legibility; taller than 10×6 so a 2×2 grid does not squash. The `PlotsConfig` model exposes the knob; helper defaults match so plain-Python callers work without Hydra.
- **D6 — default `display_tz="Europe/London"` kept, guarded by a DST-verification gate.** T3 added three gate tests exercising every time-axis helper across one spring-forward Sunday (31 March 2024) and one fall-back Sunday (27 October 2024). All three passed on first run — pandas `tz_convert("Europe/London")` handles the 1-hour spring gap and the fall-back wall-clock collision correctly, `mdates` renders without exceptions, and the heatmap's `.hour`-groupby does not double-count. No UTC-only fallback needed. The DST gate remains in the test suite as a regression guard for future pandas / matplotlib bumps.
- **D7 — ACF plot renders `lags=168` by default with labelled reference markers at lag 24 (daily) and lag 168 (weekly).** Human mandate 2026-04-20: daily seasonality is the dominant cycle in electricity demand, so the ACF must visually reflect that, not just the weekly spike. The default `lags=168` renders the full 0-168 range (capturing the daily cascade 24/48/72/96/120/144/168 plus the weekly resonance). Two labelled `axvline` markers at alpha 0.35 annotate the cycles. `plot_acf`'s own default (~39 lags for an 8760-row residual series) would miss the weekly spike entirely; the override is load-bearing for the Stage 7 SARIMAX teaching arc.
- **D8 — empirical q10-q90 quantile band.** The helper accepts a pre-computed `per_fold_errors: pd.DataFrame` indexed by fold and horizon; derivation is the caller's responsibility. Non-parametric, model-agnostic. Method (b) (`mean ± 1.96σ`) rejected because the quantile method is not much more code and is the principled choice when the fold count is small. Method (c) (statsmodels prediction interval) rejected — would couple the helper to `LinearModel.results`, violating AC-3. The uncertainty band is **not** a probabilistic prediction interval — it is an empirical characterisation of observed rolling-origin errors. Stage 10 (quantile regression) replaces it with a proper interval.
- **D9 — `evaluate(..., return_predictions=True)` accepted as a single-flag one-off.** The debt note is recorded in `src/bristol_ml/evaluation/CLAUDE.md` verbatim: a **second** boolean extension to `evaluate()` triggers a first-class `EvaluationResult` dataclass (metrics + optional predictions + optional extras) or a typed `evaluate_v2` alongside a deprecation. The tolerance here trades one bit of API surface for zero new machinery at Stage 6; the re-engineering trigger is explicit, tracked, and discoverable without re-reading this plan.
- **D10 — `NesoBenchmarkConfig.holdout_start/_end` wired via `plots.benchmark_holdout_bar`.** The helper constructs a single-fold `SplitterConfig(fixed_window=True, min_train_periods=<before-start>, test_len=<holdout window length>, step=<holdout window length>)` from the two `holdout_*` config fields, then dispatches through the existing `compare_on_holdout` three-way benchmark. Closes the Stage 4 carry-over (H-1 from the Stage 5 plan housekeeping; Stage 4 retro "Deferred" bullet #3). `compare_on_holdout` is imported lazily inside `benchmark_holdout_bar` to avoid a plots ↔ benchmarks circular-import risk; the canonical monkeypatch path for tests is `bristol_ml.evaluation.benchmarks.compare_on_holdout`.
- **D11 — append-only notebook edit, Cells 0–12 verbatim.** Human mandate 2026-04-20 reinterprets AC-5 as "a small amount of code per analysis" (each helper call is one-to-three lines), not "the total notebook shrinks". The Stage 4 notebook's Cell 11 is part of the Stage 4 pedagogical artefact; the Stage 6 value is the *new* diagnostic capability appended alongside, not the removal of the old. Implementation approach: a surgical text-level JSON splice, not `nbformat.read/write`, because `nbformat` reorders cell keys and re-encodes UTF-8 differently (e.g. `\u001B` → `0x1b` raw byte). The append script (`/tmp/append_stage6_cells.py`, not committed) finds the anchor `\n ],\n "metadata":` closing the cells array and inserts cells 13-18 as JSON at that point, leaving the preamble bytes untouched. The commit's `git diff --stat` output reads "1 file changed, 186 insertions(+), 0 deletions(-)" — additions-only, as mandated. `nbconvert --execute` was run against a `/tmp/` copy for runtime verification (1076467 bytes; only the pre-existing Cell 11 `jetTransient` schema warning from PyCharm/DataSpell); the committed notebook ships with empty outputs on the appendix cells because `nbconvert --execute` would otherwise churn Stage 4's shipped cell ordering.
- **Lazy import path for `benchmark_holdout_bar`.** The Stage 6 helper depends on `compare_on_holdout` (Stage 4) which in turn depends on `harness.evaluate` (Stage 4). If `plots.py` imported `benchmarks.py` at module scope, a downstream `from bristol_ml.evaluation.benchmarks import ...` call that transitively hits `plots.py` would cycle. The cycle is not theoretical — at T4 the first integration of the helper tripped it. The shipped fix is a function-body `from bristol_ml.evaluation.benchmarks import compare_on_holdout` inside `benchmark_holdout_bar` plus a matching `from bristol_ml.evaluation.splitter import ...` for `SplitterConfig`. Tests monkeypatch `bristol_ml.evaluation.benchmarks.compare_on_holdout` (not the plots-module rebind) because the lazy import resolves the callable from the benchmarks namespace at call time.
- **`_require_tz_aware_datetime_index` is a shared guard, not per-helper duplication.** Every time-axis helper (`residuals_vs_time`, `error_heatmap_hour_weekday`, `forecast_overlay`, `forecast_overlay_with_band`) validates at the top of the function body that the input's index is a `pandas.DatetimeIndex` and that `index.tz is not None`. The helper message names the offending parameter (`"residuals"` / `"actual"` / …) so facilitators see the error without diving into the code. The guard is deliberate — the Okabe-Ito palette plus `Europe/London` display tz only makes pedagogical sense when the timestamps are unambiguous; a tz-naive index could be anything.
- **`_SCATTER_SAMPLE_CAP = 20_000` with seeded downsampling.** `predicted_vs_actual` scatter-plots every point by default but downsamples once the input exceeds 20 k rows — a `numpy.random.default_rng(42).choice(size=20_000, replace=False)` keeps the plot deterministic run-to-run while holding its rendering cost under a second even for long training windows. The seed is a module-level constant so reproducibility is explicit.
- **Okabe-Ito index mapping by semantic role, not cycle order.** `residuals_vs_time` uses `OKABE_ITO[0]` (black) for the line and `OKABE_ITO[5]` (sky blue) for the ACF reference markers; `forecast_overlay` uses `OKABE_ITO[1:]` + `OKABE_ITO[3:]` for the prediction lines so the actual line sits visually apart; `forecast_overlay_with_band` uses `OKABE_ITO[1]` (orange) for the band fill with alpha 0.25. The mapping is documented in the module docstring so future Stage 7/8 helpers that reuse the palette know which index means what.

## Demo moment

From a clean clone (Stages 0–5 already built) with warm feature + NESO forecast caches:

```
uv sync --group dev
uv run pytest                                                              # 460 passed
uv run ruff check .                                                        # clean
uv run python -m bristol_ml.evaluation.plots --help                        # module help + palette
uv run jupyter nbconvert --execute --to notebook --inplace notebooks/04_linear_baseline.ipynb  # ~30 s; 19 cells
```

The notebook produces (in addition to the existing Stage 4 outputs):

- A 2×2 hero grid: residuals-vs-time, predicted-vs-actual scatter with 45° line, ACF with daily + weekly markers, hour-of-day × weekday heatmap centred at zero (RdBu_r).
- A three-line `forecast_overlay` call reproducing Cell 11's 48-hour view via the helper.
- A `forecast_overlay_with_band` call using `evaluate(return_predictions=True)` — the empirical q10-q90 band around the linear forecast, with per-horizon widths derived from rolling-origin fold errors.
- A `benchmark_holdout_bar` grouped bar chart of the three-way NESO comparison across MAE / MAPE / RMSE / WAPE on the `conf/evaluation/benchmark.yaml` holdout window (guarded on a warm NESO cache).

`python -m bristol_ml.evaluation.plots --help` prints the seven helper names, the active Okabe-Ito palette as hex strings, and the rcParams entries the module overrides at import time.

## Deferred

- **PACF.** External research §R4 flagged PACF as a cheap extension over ACF, differential-value at SARIMAX order-selection (Stage 7's concern). Defer; ACF with `lags=168` is enough for the Stage 6 weekly-ripple teaching moment. Revisit at Stage 7 if the SARIMAX-order picker needs it.
- **Per-horizon diagnostics.** External research §R5 and A3 both conclude "defer until a multi-horizon model ships". The Stage 6 helpers accept `horizon_h` in the predictions frame but do not pivot on it; a Stage 10/11 multi-horizon forecaster will add per-horizon `residuals_vs_time` / `forecast_overlay_with_band` variants.
- **Per-weather-regime error breakdown.** External research §R5 — domain-specific and cheap at Stage 16+, not load-bearing for the Stage 6 diagnostic surface.
- **Holiday-proximity error breakdown.** External research §R5 — a five-minute addition but adds narrative complexity not load-bearing for Stage 6.
- **Residual-histogram + Normal overlay.** Hyndman fpp3 §5.3's fifth diagnostic. Intentionally dropped because the ACF + heatmap pair already captures the structural story; adding a sixth plot would over-subscribe the 2×2 grid. Add at Stage 7 if the SARIMAX residuals look non-Gaussian enough to warrant it.
