# Stage 4 — Linear regression baseline + evaluation harness

## Goal

Land the first two models (seasonal-naive floor + OLS weather-only regression), the full point-forecast metric suite, the fold-level evaluator harness that every subsequent modelling stage will reuse, and the three-way NESO day-ahead benchmark. Fix the `Model` protocol that Stages 5, 7, 8, 10, 11 must implement. Deliver all of it behind a single-invocation `python -m bristol_ml.train` and a thin notebook, with the Stage 3 Phase 3 review carry-overs (H-1 UTC guard, H-2 `SplitterConfig` comments, H-3 `__all__` re-export) cleared as part of the work.

## What was built

- `src/bristol_ml/models/` — new layer. Public surface: the `@runtime_checkable` `typing.Protocol` `Model` (`fit`, `predict`, `save`, `load`, `metadata`) and the frozen `ModelMetadata` provenance record defined in `conf/_schemas.py` and re-exported through `models/protocol.py`; `save_joblib` / `load_joblib` atomic IO helpers in `models/io.py` (tmp + `os.replace`, mirroring `ingestion._common._atomic_write`); concrete `NaiveModel` (322 lines) and `LinearModel` (298 lines) with module CLIs `python -m bristol_ml.models.naive --help` and `python -m bristol_ml.models.linear --help`. Module guide at `src/bristol_ml/models/CLAUDE.md`.
- `src/bristol_ml/evaluation/metrics.py` — four pure `(y_true, y_pred) -> float` metrics (`mae`, `mape`, `rmse`, `wape`) as fractions per DESIGN §5.3, plus `METRIC_REGISTRY: dict[str, MetricFn]` for the harness and CLI. Shared `_coerce_and_validate` gate coerces `np.ndarray | pd.Series | list[float]` and rejects length mismatches / NaN / D8 MAPE zero-denominator / D9 WAPE zero-sum. WAPE formula pinned to Kolassa & Schütz (2007) `Σ|y − ŷ| / Σ|y|` in the docstring and a one-line inline comment. Module CLI `python -m bristol_ml.evaluation.metrics --help` prints the registered + selected metric names.
- `src/bristol_ml/evaluation/harness.py` — `evaluate(model, df, splitter_cfg, metrics, *, target_column="nd_mw", feature_columns=None) -> pd.DataFrame` (388 lines). Holds the Stage 3 carry-over **H-1 UTC guard**: tz-aware non-UTC indices raise `ValueError` before any fold is enumerated; tz-naive is permitted (splitter tolerates it). `feature_columns=None` falls back to every weather column in `assembler.WEATHER_VARIABLE_COLUMNS`, so the harness default and `LinearModel` default never disagree. Returns long-form per-fold DataFrame (`fold_index`, `train_end`, `test_start`, `test_end`, plus one float column per metric keyed by the metric callable's `__name__`). Emits one structured loguru INFO line per fold plus a completion summary (total folds, elapsed time, per-metric mean ± std).
- `src/bristol_ml/evaluation/benchmarks.py` — `align_half_hourly_to_hourly(df, *, aggregation="mean", value_columns=...)` (D4 aggregator) and `compare_on_holdout(models, df, neso_forecast, splitter_cfg, metrics, *, aggregation="mean", target_column="nd_mw", feature_columns=None) -> pd.DataFrame` (450 lines). Runs every `models` entry through `harness.evaluate`, aggregates NESO HH → hourly, and scores NESO outturn-vs-forecast on the intersection of the union of fold test-periods and the NESO hourly coverage. Returns a DataFrame indexed `[*sorted(models), "neso"]`; empty intersections raise rather than returning an empty frame. Plan D5 placement (`evaluation/benchmarks.py` rather than a peer `benchmarks/` module) recorded in the evaluation layer doc with a back-reference. Module CLI `python -m bristol_ml.evaluation.benchmarks --help`.

## Design choices made here

- **Discriminated union on `AppConfig.model` via a `type` tag, not a group name.** The plan §1 D3 sketch and DESIGN §7.3 both show a `Model` protocol but are silent on how the *selected* variant surfaces in the resolved Pydantic config. We chose a discriminated union (`ModelConfig = NaiveConfig | LinearConfig` with `Field(discriminator="type")`) because Hydra's `model=naive` / `model=linear` group override resolves to exactly one variant at a time, and the `type: Literal["naive"]` / `type: Literal["linear"]` tag written into each YAML group file lets Pydantic v2 pick the right model without reflection. The alternative — a `ModelsGroup` with `naive: NaiveConfig | None = None, linear: LinearConfig | None = None` — was in the plan §7 "Files" list, but it would permit pathological multi-variant configs that Hydra itself cannot produce, so the tagged-union shape is the tighter contract. The `train` CLI dispatches via `isinstance(model_cfg, NaiveConfig)` which is exhaustive over the union.
- **Harness `feature_columns=None` default reuses `assembler.WEATHER_VARIABLE_COLUMNS`.** The plan said "all weather float32 columns from `assembler.OUTPUT_SCHEMA`". The practical resolution is to enumerate from the existing `WEATHER_VARIABLE_COLUMNS: tuple[tuple[str, pa.DataType], ...]` constant at the top of `assembler.py` rather than re-filter the schema at runtime — same data, cleaner import. The implication is load-bearing: when Stage 5 adds calendar columns, the harness default follows automatically because `LinearModel._resolve_feature_columns` reads the same constant. Tests in `test_harness.py` pin this by asserting the default `feature_columns` on a synthetic frame match `[name for name, _ in WEATHER_VARIABLE_COLUMNS]`.
- **`compare_on_holdout` scores NESO on the union of per-model fold test-periods, not on a separate `holdout_start/_end` config pair.** The plan §5 signature advertised `splitter_cfg` + `metrics` and the `NesoBenchmarkConfig` scheme in `conf/_schemas.py` carried a `holdout_start` / `holdout_end` pair, but in practice every fold's test-period is already derivable from the harness output; a second config surface for the holdout window would only introduce drift between "which folds the models were evaluated on" and "which span NESO was scored against". So the benchmark helper computes `holdout_start = min(fold_spans.test_start)`, `holdout_end = max(fold_spans.test_end)` internally and masks NESO by that span. `NesoBenchmarkConfig.holdout_start/_end` remain in the schema for a future Stage 6 richer-diagnostics flow where an explicit holdout window is meaningful; today nothing consumes them. Surfaced as a soft deviation from the plan §5 signature; the narrower `compare_on_holdout(... splitter_cfg, metrics ...)` version is what shipped.
- **Naive-model state is the full `(features, target)` frame, not a compact lookup table.** Plan T3 proposed "store the training-end 168 hours of target plus index". In practice, `same_hour_last_week` can be asked about any test timestamp, and joining on `target.index - Timedelta(hours=168)` is cleaner when the training frame already sits in memory. The save/load round-trip (joblib of the `NaiveModel` instance) pickles the pandas `Series` directly and costs a few tens of KB per year of training data — well inside any reasonable budget. The plan's `ValueError` guard on missing look-back rows is preserved at `predict()` time.
- **Lazy-import idiom replicated for `bristol_ml.models`.** `models/__init__.py` uses the same `if TYPE_CHECKING` + `__getattr__` pattern as `bristol_ml.evaluation/__init__.py`, keeping the top-level `python -m bristol_ml` scaffold invocation cheap. Concrete model classes are *not* re-exported at the package level — `from bristol_ml.models import NaiveModel` is intentionally a `ModuleNotFoundError`; `from bristol_ml.models.naive import NaiveModel` is the only import path. This keeps statsmodels import-time cost off every `bristol_ml` import.
- **Discriminated-union `type` literal pinned to the Hydra group filename.** `NaiveConfig.type: Literal["naive"] = "naive"` matches `conf/model/naive.yaml`; `LinearConfig.type: Literal["linear"] = "linear"` matches `conf/model/linear.yaml`. YAML group files write `type: naive` / `type: linear` into their payload explicitly (Hydra does not inject the discriminator); the test `test_model_config_discriminates_on_type_tag` pins this in both directions.
- **Train CLI is in-process testable.** `_cli_main(argv: Iterable[str] | None = None) -> int` accepts an explicit `argv` list, letting five of the six `test_train_cli.py` tests drive the CLI via a direct in-process call with `capsys` (fast). One additional test runs a `subprocess.run` against `python -m bristol_ml.train --help` to exercise the `__main__.py` wiring. The pattern is documented for future Stage 5+ CLI tests.
- **Notebook splitter override narrows the window, not the stride.** The shipped `conf/evaluation/rolling_origin.yaml` defaults are tuned for the full-year training window the CLI expects. The notebook overrides both `evaluation.rolling_origin.min_train_periods=720` (30 days) and `step=168` (weekly) so the demo finishes in ~8 s with the 2023-only warm cache; the CLI override set is narrower (step only) because CI data provisioning is not a concern there. Documented inline in the notebook's Cell 2 markdown.
- **NESO forecast schema pinned from the live CKAN response.** Following the plan §10 risk-register guidance, the first VCR cassette recorded the real `datastore_search` response and `OUTPUT_SCHEMA` was derived from that cassette's columns. The source column names are CKAN-native `CamelCase_With_Underscore` (`Demand_Forecast`, `Demand_Outturn`, `APE`, `Date`, `Settlement_Period`); the ingester renames them to snake_case + `_mw`/`_pct` suffixes in `fetch()` to match the project convention. `IDENTITY_RAW_COLUMNS` + `OUTPUT_SCHEMA` + the rename map are all module-level constants, so a schema change upstream surfaces as a test failure against the cassette rather than a silent column drop.
- **H-1 UTC guard lives in the harness, not the splitter.** Per plan T0's note and the Stage 3 Phase 3 review: the splitter accepts only `n_rows` and cannot inspect the DataFrame's index, so it cannot enforce tz-aware-UTC. `evaluate()` is the first function in the evaluation path that sees the frame, so that is where the guard belongs. Tz-naive is permitted (the splitter's semantics do not change); `Europe/London` or any other non-UTC tz raises before any fold is enumerated. Test: `test_harness_rejects_non_utc_index`.

## Demo moment

From a clean clone (Stages 0-3 already built) with warm feature and forecast caches:

```
uv sync --group dev
uv run pytest                                                              # 360 passed
uv run python -m bristol_ml.train --help                                   # CLI help
uv run python -m bristol_ml.train                                          # default (linear)
uv run python -m bristol_ml.train model=naive                              # swap via Hydra override
uv run jupyter nbconvert --to notebook --execute notebooks/04_linear_baseline.ipynb --output /tmp/04_test_run.ipynb  # ~8 s
```

The notebook produces the `statsmodels` `results.summary()` block, the per-fold metric tables for both models, the three-way benchmark (when the NESO forecast cache is warm), and the residual + 48-hour overlay figure.

## Deferred

- **Persisting per-fold predictions.** D10 explicitly defers this. A future Stage 6 (enhanced evaluation + visualisation) will add a `predictions_path: Path | None = None` kwarg on `evaluate()`; nothing in Stage 4 writes per-fold predictions to disk.
- **`skops.io` adoption for secure artefacts.** D6 pinned joblib for Stage 4 with a one-line comment in `models/io.py` pointing the Stage 9+ upgrade path. Revisit when the registry lands.
- **`ingestion._neso_dst` extraction.** The settlement-period → UTC algebra is now duplicated in `neso.py` and `neso_forecast.py`; a follow-up refactor (`_autumn_fallback_dates`, `_spring_forward_dates`, `_settlement_to_utc`) into `_common` is noted in `ingestion/CLAUDE.md`. Out of scope for Stage 4 because lifting would edit the Stage 1 surface.
- **`NesoBenchmarkConfig.holdout_start/_end` consumers.** The schema fields exist and are validated, but `compare_on_holdout` currently derives the holdout window from fold test-periods. A Stage 6 richer-diagnostics flow (fixed-window retrospective comparison outside the rolling-origin harness) is the natural consumer; the soft deviation is noted above.
- **"Beat NESO" as a success gate.** D11: not a Stage 4 goal. The notebook frames the expected loss-to-NESO as the pedagogical setup for Stage 5 calendar features; no CI assertion enforces relative performance.
- **Pre-2021 benchmark coverage.** The NESO day-ahead forecast archive starts April 2021; the three-way table is silently restricted to the overlap of fold test-periods and forecast coverage. Callers operating on 2018-2020 feature tables will see an `intersection` `ValueError` from `compare_on_holdout`; documented in the benchmark's docstring and in `src/bristol_ml/evaluation/CLAUDE.md`.
- **Known drift, surfaced for the human (do not fix from within Stage 4):**
  - `docs/intent/DESIGN.md` §6 layout tree — not updated here (deny-tier for the lead). Stages 1-4 additions (`models/`, `evaluation/{metrics,harness,benchmarks}.py`, `ingestion/neso_forecast.py`, `train.py`, five new `conf/` groups) are missing; accumulate for a batched human edit.
  - DESIGN §8 "Open-Meteo UKV 2km" claim remains incorrect (from Stage 2); unchanged here.
