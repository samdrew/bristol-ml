# Stage 9 — Model registry

## Goal

Ship the filesystem-backed model registry that turns "which model is best" into a single CLI command. Every model family from Stages 4, 7, and 8 — `NaiveModel`, `LinearModel`, `SarimaxModel`, `ScipyParametricModel` — registers through the same four-verb interface without any change to the model classes themselves (intent AC-2). The `python -m bristol_ml.registry list` leaderboard is the Stage 9 Demo moment (intent §Demo moment); the retrofit is the structural proof that the `Model` protocol's five members are a sufficient contract; the four-verb cap (`save`, `load`, `list_runs`, `describe`) is the load-bearing constraint under AC-1.

The central design question — *how small can the registry's public surface be while still enabling the leaderboard, the retrofit, and the Stage-12 load-by-name handoff?* — resolved pre-implementation via D12 (four verbs in `__all__`, structurally enforced), D17 (save path wired into the `train` CLI via a *new* harness function `evaluate_and_keep_final_model` rather than a second boolean on `evaluate`), and D10 (MLflow graduation is documented **and** tested via a PyFunc adapter under `tests/integration/`, without promoting MLflow to a runtime dependency). That trio kept AC-1 honest, the harness API-growth rule intact (`evaluation/CLAUDE.md` H5), and the "mechanical migration" claim falsifiable rather than aspirational.

## What was built

- **T1 — Module scaffold + on-disk helpers + git-SHA helper.** `src/bristol_ml/registry/__init__.py` with `__all__ = ("save", "load", "list_runs", "describe")` (noqa RUF022 to keep verb-chronological order) and `DEFAULT_REGISTRY_DIR = Path("data/registry")`. Private modules: `_fs.py` hosts `_build_run_id(model_name, fit_utc) -> str` (minute-precision UTC ISO-8601 regex `^[a-z][a-z0-9_.-]*_\d{8}T\d{4}$` per D3 — human Ctrl+G amendment dropped seconds from the original `YYYYMMDDTHHMMSS`), `_run_dir(registry_dir, run_id) -> Path`, and `_atomic_write_run(registry_dir, run_id, artefact_writer, sidecar) -> None` using a `.tmp_{uuid4.hex}/` staging directory renamed via `os.replace` (D5 mirrors the two existing atomic-write sites in the codebase). `_git.py` hosts `_git_sha_or_none() -> str | None` running `git rev-parse HEAD` with a 2 s subprocess timeout (D13 / AC-3); `None` is a legitimate return value outside a git tree or on a shallow clone. `_schema.py` hosts `SidecarFields` and `MetricSummary` TypedDicts (D4 — structural type, not a Pydantic model, so the registry does not couple to a validated-read path Stage 9 does not yet need). T1 tests: 10 gates (structural AC-1 cap, run-ID regex, UTC-naive rejection, git-SHA in/out of tree, atomic-write layout + last-write-wins + staging cleanup on error, `DEFAULT_REGISTRY_DIR` constant).

- **T2 — `save()` + the four AC-2 save tests.** `registry.save(model, metrics_df, *, feature_set, target, registry_dir=None) -> run_id` constructs the sidecar from `model.metadata`, caller-supplied kwargs, `_git_sha_or_none()`, and per-metric summaries (`_summarise_metrics` computes `{mean, std, per_fold}` per non-timestamp column). Calls `model.save(run_dir / "artefact" / "model.joblib")` via the existing protocol (D9 — no model-code changes). JSON written with `json.dumps(..., indent=2, allow_nan=True, ensure_ascii=False)` — `allow_nan=True` preserves `ScipyParametricModel`'s `float("inf")` covariance entries (D4 + R3). `_dispatch.py` introduces the registry-local type dispatcher (H4 — *not* a third shared dispatcher site beside `harness.py::_build_model_from_config` and `train.py::_cli_main`) with two inverse look-ups: `_CLASS_NAME_TO_TYPE` keyed on class `__name__` so `_NamedLinearModel` (the Stage 5 `train.py` internal wrapper) maps to `"linear"` without a circular import, and `_TYPE_TO_CLASS` for the load path. T2 tests: four AC-2 `save`-path tests across all four model families plus AC-3 explicit-kwarg rejection and the git-SHA auto-capture.

- **T3 — `load()` + AC-2 round-trip.** `registry.load(run_id, *, registry_dir=None) -> Model` reads the sidecar, dispatches on the `type` field via `_dispatch.class_for_type`, and delegates to the concrete class's `Model.load(path)` classmethod. `FileNotFoundError` on missing sidecar or artefact, `ValueError` on unknown `type` string. `_NamedLinearModel` case: returns base `LinearModel`; the sidecar's `name` field preserves the dynamic name as the source of truth for readers, but is not re-applied to the loaded instance (D16 cut — dynamic-name round-trip would require `train.py` state that the registry is deliberately ignorant of). T3 tests: round-trip `predict` agreement to `atol=1e-10` per model family + `_NamedLinearModel`-returns-base-class guard + missing-run and unknown-type error paths.

- **T4 — `list_runs()` + `describe()` + AC-4 perf gate.** `registry.list_runs(*, target=None, model_type=None, feature_set=None, sort_by="mae", ascending=True, registry_dir=None) -> list[dict]` scans `registry_dir` with a single `iterdir()`, skips `.tmp_*` staging directories, reads each `run.json`, applies the three D7 exact-match filters, and sorts via `_sort_key(sidecar, metric_name, ascending=...)` returning `(missing_flag, value)` so runs missing the sort metric land last in both directions. `describe(run_id, ...) -> dict` is a one-sidecar read with `FileNotFoundError` on miss. T4 tests: 12 gates including the AC-4 `< 1 s on 100 runs` perf gate, three filter dimensions, four sort scenarios (default, descending, alternative metric, missing metric sorts last), staging-directory skip, empty-registry return, and missing-run error.

- **T5 — CLI + harness final-model extension + train-CLI wiring.** `src/bristol_ml/registry/__main__.py` implements `python -m bristol_ml.registry {list,describe}` via `argparse` (same idiom as `train.py`, D16). `list` prints a leaderboard table with plain `str.format` padding (no `tabulate` dep); `describe` pretty-prints one sidecar as indented JSON with `allow_nan=True`. `--registry-dir` override on both subcommands. `src/bristol_ml/evaluation/harness.py` gains `evaluate_and_keep_final_model(model, df, splitter_cfg, metrics, *, target_column="nd_mw", feature_columns=None) -> tuple[pd.DataFrame, Model]` (D17) — delegates to `evaluate()` so the metric-computation contract stays at one function; respects the `evaluation/CLAUDE.md` H5 API-growth rule (no second boolean flag on `evaluate`). `src/bristol_ml/train.py` gains `--registry-dir` and `--no-register` CLI flags; on the default path it calls `evaluate_and_keep_final_model` and then `registry.save(final_model, metrics_df, feature_set=..., target=...)`, printing `Registered run_id: {run_id}`. **No re-fit on full training data** — the final-fold model already in harness memory is what gets saved (D17 rationale: re-fit would add a second training pass per CLI invocation for no named requirement, and the leaderboard metric row is the rolling-origin cross-fold summary so the final-fold artefact is an honest representative of what the metrics describe). T5 tests: 9 gates including `list` / `describe` / `--help` via subprocess, `evaluate_and_keep_final_model` tuple shape, and the train-CLI registration + `--no-register` paths.

- **T6 — Layer documentation + AC-5 tests.** `docs/architecture/layers/registry.md` (D18) covers the full Stage 9 contract: on-disk directory-tree diagram, sidecar JSON schema annotated field-by-field, the four-verb Python API with full signatures, the `list_runs` vs `list` rename rationale, the type-dispatch table, `train.py` integration story, the "what artefact is saved" rationale for the final-fold choice (R2), the Graduation-to-MLflow subsection (D10 — names the PyFunc adapter pattern, the three migration-contract points from domain §R5, and the `src/` vs `tests/` housing rationale), the joblib / `skops.io` deferral note pointing at Stage 12 (D14), NFR gates, upgrade seams, and a module inventory. T6 tests: `test_registry_layout_documentation_exists` (layer doc > 50 lines and contains `"Graduation to MLflow"` + `"skops"` markers) + `test_registry_run_json_is_hand_parseable` (real registered run parses with the stdlib `json` module with every `SidecarFields` key present, correctly typed, `indent=2` formatting evidence, no `\u` escapes from `ensure_ascii=False`).

- **T7 — MLflow PyFunc adapter (test-only) + D10 round-trip test.** `tests/integration/mlflow_adapter.py` — `RegistryPyfuncAdapter(mlflow.pyfunc.PythonModel)` implements `load_context` (reads `context.artifacts["registry_run"]`, derives `registry_dir` and `run_id` from the path, delegates to `registry.load(run_id, registry_dir=registry_dir)`) and `predict` (forwards `model_input` to the wrapped `Model.predict`). `package_run_as_pyfunc(run_id, dst, *, registry_dir)` helper calls `mlflow.pyfunc.save_model(path=str(dst), python_model=adapter, artifacts={"registry_run": str(run_dir)})`. Adapter lives under `tests/integration/` (not `src/`) so the public registry surface stays at four verbs (AC-1) and MLflow stays a test-only dependency (zero runtime import footprint). `tests/integration/test_registry_mlflow_adapter.py` round-trips a registered `NaiveModel`: `registry.save` → `package_run_as_pyfunc` → `mlflow.pyfunc.load_model` → `predict`, asserting `numpy.allclose(..., atol=1e-10)` element-wise against the wrapped model's direct `predict` output. `mlflow>=2.14,<3` added to the `dev` dependency group; `pytest.importorskip("mlflow")` gates the test so lean installs without `--group dev` still pass the rest of the suite. D10 upgrade (from the plan's original "document the adapter" to "document **and** test the adapter") was a human Ctrl+G amendment 2026-04-23 — a test is a falsifier for the "mechanical migration" claim, a paragraph of prose is not.

- **T8 — Stage hygiene.** This retro; `CHANGELOG.md` Stage 9 bullets under `[Unreleased]`; `docs/architecture/layers/models.md` updated per housekeeping H-3 (`skops.io` upgrade-seam row moved Stage 9 → Stage 12, D14; "Cross-version load compatibility" open question edited to reflect the Stage 9 `git_sha` capture that landed; `ModelMetadata.hyperparameters` open question updated to note Stage 9 stores the bag verbatim in `run.json`); `src/bristol_ml/models/io.py` module docstring updated to name Stage 12 as the `skops.io` trigger (D14); `src/bristol_ml/models/CLAUDE.md` security-note paragraph re-worded for the Stage-12 inflection point; H-2 verified (Stage 8 retro's "Next" pointer accurately predicts Stage 9's scope apart from the `skops.io` prediction which is now documented as deferred to Stage 12 — documentation, not a retro rewrite, per the "plan updates happen in-place" rule); H-1 flagged for the human at PR review (DESIGN §6 batched edit covering Stages 1–9; deny-tier for the lead per `CLAUDE.md`); H-4 (dispatcher-duplication ADR) re-deferred — Stage 9 added `registry/_dispatch.py` as a *registry-local* dispatcher, deliberately not a third shared site; the Stage 7/8 ADR-worth question about `_build_model_from_config` + `train.py::_cli_main` duplication carries forward unchanged to Stage 11 or a dedicated housekeeping stage. Plan moved from `docs/plans/active/09-model-registry.md` to `docs/plans/completed/09-model-registry.md` as the final commit action.

## Design choices made here

Plan §1 D1–D19 and NFR-speed / NFR-transparency plus H-1..H-4. The nineteen decisions plus the four housekeeping carry-overs were approved at Ctrl+G review 2026-04-23 with two human amendments (D3 minute precision; D10 upgrade from document-only to documented-and-tested adapter). Key points and why they bind:

- **D1 — Flat layout `data/registry/{run_id}/` with `artefact/` + `run.json`.** One `os.listdir` + one `json.load` per run for the leaderboard query; avoids MLflow's one-file-per-key directory-explosion pathology. Subdirectory per run contains joblib's occasional multi-file output. Nested layout (`target/model/date/`) would be easier to browse but forces multi-level traversal for any cross-run query.
- **D2 — One entry per `save()` invocation; last-write-wins on same-run-id collision.** Matches every industry precedent at research §R6 (MLflow UUID-per-invocation, W&B run-id, DVC, BentoML `{name}:{datetime}`). Identity semantics beyond "same filesystem name" would be scope no AC asks for.
- **D3 (human-amended) — `{model_name}_{YYYYMMDDTHHMM}` run IDs.** Minute precision. Typeable at the CLI during a live demo; sortable lexicographically so default list ordering is free. Same-minute collisions at single-author pace are rare and handled by D2 last-write-wins (R7). Moving to second precision is a no-schema-change graduation if multi-worker training ever raises the collision rate.
- **D4 — JSON sidecar, `indent=2, allow_nan=True, ensure_ascii=False`.** `allow_nan=True` preserves `ScipyParametricModel`'s `float("inf")` covariance entries (R3). JSON needs no new dependency; YAML would add `pyyaml`; SQLite would be a rewrite for no named requirement.
- **D5 — Temp-dir-then-rename atomic save.** Mirrors the codebase's two existing atomic-write sites (`ingestion/_common.py::_atomic_write`, `models/io.py::save_joblib`). A crash mid-write leaves the previous run intact or absent — never partial. Eliminates MLflow's "dead-run" pathology (sidecar without artefact) at the cost of four lines of Python.
- **D6 — Defer feature-table content hash.** Git SHA + feature-set name is enough provenance at single-author scale. Promote if Stage 17 (price-target models) needs deterministic dataset identity for reproducibility claims.
- **D7 — Three exact-match filters (`target`, `model_type`, `feature_set`).** The minimum common interface downstream consumers need; no wildcards, no date range at Stage 9. Adding a filter means adding a kwarg, a test, and a CLI flag — the cost is bounded.
- **D8 — `sort_by="mae"` ascending default; missing metrics sort last.** MAE-ascending is the defensible Demo-moment default from research §R9; making it the built-in default keeps `python -m bristol_ml.registry list` a one-word invocation.
- **D9 — Registry calls `Model.save(path)`; no protocol change.** AC-2 ("without code changes to the model itself") is satisfied by construction. The registry duplicates neither serialisation logic nor per-family save knowledge — the type dispatcher is the only registry-held model knowledge, and it is a single four-entry dict literal in `_dispatch.py`.
- **D10 (human-upgraded) — Document AND test the MLflow PyFunc graduation path.** A test proves the "mechanical migration" claim is falsifiable. If MLflow 3.x breaks PyFunc semantics, the red test flags drift at the version-bump boundary rather than silently in production. Test-only housing keeps AC-1's four-verb cap intact.
- **D11 — `data/registry/` gitignored via the existing `data/*` rule; no portable export tool.** Nothing to add to `.gitignore`; portable export is scope no AC asks for.
- **D12 — Public surface is four verbs, enforced structurally.** AC-1 is load-bearing; `__all__` pinned to a tuple so `test_registry_public_surface_does_not_exceed_four_callables` is a one-line assertion. `list_runs` is the exported symbol; `list` is the CLI subcommand and the layer-doc verb. A fifth verb requires retiring one first.
- **D13 — Git SHA captured inside the registry, not by the model.** `_git_sha_or_none()` is a private helper with a 2 s subprocess timeout, `check=False`, returning `None` on any failure. AC-3 is satisfied without modifying the four model classes (AC-2 contract preserved).
- **D14 — Keep joblib; defer `skops.io` to Stage 12.** Stage 9 only loads artefacts the training author wrote themselves. Stage 12 (serving) is the first consumer that may load from a path not controlled by the training author — the correct inflection point for the security upgrade. One-line docstring edit and one layer-doc row move is the cheapest way to record the deferral (H-3).
- **D15 — Metrics summarised from caller-supplied per-fold DataFrame.** `_summarise_metrics` computes `{mean, std, per_fold}` per non-timestamp column at save time. The harness is unchanged (respects `evaluation/CLAUDE.md` H5 — no second boolean on `evaluate()`).
- **D16 — New `__main__.py` with argparse; no new Hydra config group.** Same pattern as `train.py`; `--registry-dir` flag covers tests and throwaway demos. `python -m bristol_ml.registry` stays a zero-dep CLI.
- **D17 — Wire `train.py` via `evaluate_and_keep_final_model`, not re-fit-on-full-data.** The leaderboard metric is the cross-fold summary; the artefact is the final-fold model that honestly represents that measurement. Two `train.py` flags (`--registry-dir`, `--no-register`) cover test and notebook use.
- **D18 — New layer doc at `docs/architecture/layers/registry.md`; non-empty + Graduation-to-MLflow + skops-deferral gates.** One file, three structural assertions, nothing new to discover.
- **NFR-speed — `list_runs` < 1 s over 100 runs.** Quantifies intent "instantaneous" with an order-of-magnitude CI headroom above the plausible tens-of-milliseconds local-filesystem walk.
- **NFR-transparency — `cat run.json` prints parseable JSON.** `indent=2`, no binary escape, no YAML dependency (AC-5).

## Surprises captured during implementation

- **`_NamedLinearModel` round-trip could not be circular.** The Stage 5 `train.py` internal wrapper `_NamedLinearModel` overrides `metadata.name` but is defined in `train.py`. Importing it from `bristol_ml.registry._dispatch` would create a circular import (`train.py` imports `registry`, `registry._dispatch` imports `train._NamedLinearModel`). Resolved by keying `_CLASS_NAME_TO_TYPE` on `type(model).__name__` rather than on the class object itself — the dispatcher never holds a direct reference to the `train.py` wrapper. The round-trip then falls through to plain `LinearModel.load` (which is fine, because `_NamedLinearModel.save` delegates to its inner `LinearModel.save`, so the artefact is a base-class `LinearModel` on disk). Plan D16 cut the dynamic-name round-trip explicitly, so the sidecar's `name` field is the source of truth for the human-readable name and nothing on the loaded-instance side needs to re-apply it.
- **Train-CLI test fixtures polluted `data/registry/`.** The pre-Stage-9 `test_train_cli.py` fixture `warm_feature_cache` warmed the feature cache into `tmp_path` but did not redirect `DEFAULT_REGISTRY_DIR`. Once `train.py` started calling `registry.save(...)` by default, running the existing train-CLI tests wrote real registry runs to the repo's `data/registry/`. Resolved by `monkeypatch.setattr(registry, "DEFAULT_REGISTRY_DIR", tmp_path / "registry")` inside the `warm_feature_cache` fixture. `rm -rf data/registry/` cleaned up the artefacts from the development-time runs before commit.
- **`test_registry_verbs_are_stubs_until_later_tasks` required three updates.** The T1 stub test asserted that every verb raised `NotImplementedError`; each subsequent task (T2 `save`, T3 `load`, T4 `list_runs` + `describe`) chipped one assertion off. By the end of T4 the test was renamed `test_registry_verbs_are_all_implemented` and verified callability — the simplest test that still pins "every named verb is present and reaches production code". Plan §6 anticipated this stepdown; it read less awkward in the commit history than "delete and re-add".
- **MLflow's PyFunc `predict` signature evolved.** MLflow 2.x added an optional `params` kwarg to `PythonModel.predict` that earlier 2.x releases did not have. The adapter's `predict(self, context, model_input, params=None)` signature accepts both worlds; not specifying `params` in the adapter would have worked against the installed 2.22.4 but would have been a future-compat foot-gun. Test passes against 2.22.4; the `mlflow>=2.14,<3` pin documents the supported range.
- **`mlflow-skinny` vs `mlflow` at install time.** `uv sync --group dev` with `mlflow>=2.14,<3` resolved to `mlflow-skinny==2.22.4`; `mlflow.pyfunc.save_model` + `load_model` both work on skinny so no `mlflow[all]` extra was needed.

## What did not change

- **`bristol_ml.models.protocol.Model`** — the five-member protocol. AC-2 "without code changes to the model itself" is satisfied by construction; the registry consumes `save`, `load`, and `metadata` unchanged.
- **The four model classes** (`NaiveModel`, `LinearModel`, `SarimaxModel`, `ScipyParametricModel`). Zero body changes; zero import additions. The only adjacency is `_CLASS_NAME_TO_TYPE` in the registry's `_dispatch.py`, which holds class names, not imports of the families themselves beyond what `_TYPE_TO_CLASS` already requires.
- **`evaluation/harness.py::evaluate`** — signature and body unchanged. The new `evaluate_and_keep_final_model` lives alongside it (H5 API-growth rule).
- **`conf/_schemas.py`, `conf/config.yaml`, `conf/registry/`** — no Pydantic schema, no Hydra config group, no `AppConfig` field (D17 cut).
- **`pyproject.toml [project].dependencies`** — MLflow went into `dev` only.
- **`docs/intent/DESIGN.md` §6 layout tree** — H-1 batches this for the human at PR review (deny-tier for the lead).
- **ADR 0002 filesystem-registry-first** — the Stage 0 ADR's thesis ("joblib + sidecar is enough until Stage 9") held; Stage 9 operationalised it. The ADR is not superseded by Stage 9 — it is *fulfilled*.
- **`_NamedLinearModel.load` at `train.py:405`** — its pre-existing `NotImplementedError` is out-of-scope per §Out of scope; a future `train.py` cleanup can address it.
- **Dispatcher-duplication ADR** — H-4 re-deferred. `registry/_dispatch.py` is registry-local (the registry's verbs need type knowledge intrinsically); it is *not* the third site of a general-purpose model factory. The `_build_model_from_config` + `train.py::_cli_main` duplication carries forward unchanged.

## Open questions

- **When to hash the feature table.** D6 defers. Stage 17's price-model provenance claim may force the issue; the migration point is one new `feature_table_sha256` field in `SidecarFields` plus one test.
- **When to add a SQLite index alongside `run.json`.** Flat scan satisfies NFR-speed at the 100-run scale with tenfold headroom. Thousands of runs (grid search, repeated notebook re-runs) would eventually cross into noticeable latency. The migration point is a SQLite index *alongside* the sidecars — not a wholesale database rewrite.
- **When to expose a fifth verb.** Retire one first. AC-1 is load-bearing. A hypothetical `promote(run_id, stage)` for dev/staging/prod semantics is the most likely Stage-18 ask; the layer-doc edit would accompany that decision.
- **Dispatcher-duplication ADR** (H-4 carry-over from Stages 7–9). The Stage 7 Phase 3 review filed it as candidate ADR B1; Stage 9 did not expand its scope. The Stage 9 `_dispatch.py` is explicitly registry-local and deliberately not unified with the harness + train-CLI inline dispatchers. Revisit at Stage 11 (second NN family arriving) or via a dedicated housekeeping stage. ADR filename earmarked: `0004-model-dispatcher-consolidation.md`.
