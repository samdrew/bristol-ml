# Stage 5 — Calendar features

## Goal

Land the second feature set (`weather_calendar`) alongside the Stage 3 `weather_only` set, without touching `LinearModel`, `harness.evaluate`, `compare_on_holdout`, `SplitterConfig`, or any of the Stage 4 plumbing. Encode the four classical calendar effects — hour-of-day, day-of-week, month-of-year, and GB bank holidays — as one-hot dummies on top of the hourly feature table, and show that a single Hydra group-swap (`features=weather_calendar`) is all it takes to close most of the gap to the NESO day-ahead benchmark that Stage 4 left open.

## What was built

- `src/bristol_ml/ingestion/holidays.py` — GB bank-holidays ingester targeting `https://www.gov.uk/bank-holidays.json` under the UK Open Government Licence v3.0. Public `fetch(config, *, cache=CachePolicy.AUTO)` + `load(path)` mirror the ingestion layer contract. Consolidated parquet across all three UK divisions (`england-and-wales`, `scotland`, `northern-ireland`); the persisted file is policy-agnostic so a future regional stage does not need to re-ingest. Duplicate `(date, division)` keys raise as upstream drift; empty feeds raise rather than persist an empty cache. On-disk schema (`date`, `division`, `title`, `notes`, `bunting`, `retrieved_at_utc`) pinned in `OUTPUT_SCHEMA` and documented in `src/bristol_ml/ingestion/CLAUDE.md`. Reuses `_common._retrying_get`, `_respect_rate_limit`, `_atomic_write` verbatim. Module CLI `python -m bristol_ml.ingestion.holidays [--cache auto|refresh|offline]`. Cassette at `tests/fixtures/holidays/cassettes/holidays_refresh.yaml` (~31 kB; one GET of the full feed).
- `src/bristol_ml/features/calendar.py` — pure derivation of the 44 calendar columns. `derive_calendar(df, holidays_df) -> pd.DataFrame` appends 44 `int8` columns: 23 hour-of-day one-hots (`hour_of_day_01` … `hour_of_day_23`; `hour_of_day_00` is the reference), 6 day-of-week one-hots (`day_of_week_1` … `day_of_week_6`; Monday = 0 is the reference), 11 month one-hots (`month_02` … `month_12`; January is the reference), plus four holiday flags: `is_bank_holiday_ew`, `is_bank_holiday_sco`, `is_day_before_holiday`, `is_day_after_holiday`. `is_weekend` is deliberately **not** emitted — perfect collinearity with the day-of-week one-hots per external research §R5, pinned by a module-level assertion and a runtime guard. Emits one structured loguru INFO line per call and a single WARNING when pre-window rows are zero-filled (plan D-6). Module CLI `python -m bristol_ml.features.calendar --help`.
- `src/bristol_ml/features/assembler.py` extended with a second schema and a second orchestrator, leaving the Stage 3 surface byte-compatible. New public constants: `CALENDAR_VARIABLE_COLUMNS` (re-exported from `features.calendar`), `CALENDAR_OUTPUT_SCHEMA: pa.Schema` (55 columns = `OUTPUT_SCHEMA.names` (10) + `CALENDAR_VARIABLE_COLUMNS` (44) + `holidays_retrieved_at_utc` (1)). New public readers / orchestrators: `load_calendar(path) -> pd.DataFrame` (schema-validating read; refuses both missing and extra columns) and `assemble_calendar(cfg, *, cache="offline") -> Path` (one-shot orchestrator composing NESO demand + weather + holidays into a Stage 5 feature-table parquet). The weather-only schema is preserved as an exact prefix of the calendar schema (positions 0..9), so downstream code that reads only the weather columns composes with either frame by column-name selection. See §"Design choices made here" for why `assemble_calendar` duplicates the NESO/weather/resample/`build` composition inline rather than delegating to `assemble()`.
- `src/bristol_ml/train.py` gained `_resolve_feature_set(cfg) -> tuple[FeatureSetConfig, _LoadFn, tuple[str, ...]]`. The function resolves the active Hydra group-swap to the persisted `FeatureSetConfig`, the assembler loader (`load` vs `load_calendar`), and the resolved feature column names. Both-or-neither populated raises `ValueError`. The CLI feeds the resolved columns into both `LinearConfig.feature_columns` (unless the user pinned them via Hydra) and `harness.evaluate(..., feature_columns=...)` / `compare_on_holdout(..., feature_columns=...)` so the linear regressor, evaluator, and three-way benchmark never disagree on "what are the features?". A `_NamedLinearModel` wrapper substitutes `metadata.name` with `linear-ols-<feature-set>` at fit time, preserving the Stage 4 directive to keep `LinearModel` unchanged.
- `conf/_schemas.py` — new `HolidaysIngestionConfig` (URL, divisions, cache location, request timeout, retry / rate-limit knobs with `min_inter_request_seconds = 0.0` default per `holidays.py` §"Rate limit"); `IngestionGroup.holidays` added; `FeaturesGroup.weather_calendar: FeatureSetConfig | None = None` added alongside the existing `weather_only` slot. Both feature-set slots share the Stage 3 `FeatureSetConfig` Pydantic model — no schema drift.
- `conf/features/weather_calendar.yaml`, `conf/ingestion/holidays.yaml` — two new Hydra group files with `# @package ...` headers matching their container fields. `conf/config.yaml` `defaults:` list re-wired from package-override form (`- features/weather_only@features.weather_only`) to group-swap form (`- features: weather_only`), so `features=weather_calendar` at the CLI swaps which feature file loads. Runtime effect: exactly one of `cfg.features.weather_only` / `cfg.features.weather_calendar` is non-`None` per run.

## Design choices made here

- **`hour_of_day` one-hots read the UTC hour, not `Europe/London` local time** (human mandate 2026-04-20, overriding the T3 first implementation). Rationale: every UTC calendar day has exactly 24 rows, spring-forward and autumn-fallback Sundays included; using local hour would make the dummies rank-deficient on DST-change days (46 vs 50 settlement periods in NESO terms) and introduce a subtle discontinuity in the `hour_of_day_*` column mean on those dates. Day-of-week and month continue to read from the `Europe/London` local components (plan D-4 / D-7) because those are the effects the GB load responds to — a Sunday in UTC is still a Sunday in London for load-profile purposes. The division is documented in `src/bristol_ml/features/CLAUDE.md` and pinned by `test_hour_of_day_reads_utc_not_local`.
- **`assemble_calendar` duplicates the NESO/weather/resample/`build` composition inline rather than delegating to `assemble()`.** The plan §6 Task T4 originally described delegation; the Hydra group-swap refactor in T1 made it impossible. Under group-swap semantics, when the user invokes `features=weather_calendar` the `cfg.features.weather_only` attribute is `None`, and `assemble()`'s first guard raises with a message naming the missing field. The duplication (~25 lines) is acknowledged as a deliberate short-term cost in the plan's Implementation Findings section and in the function's own docstring. Factoring a shared `_compose_weather_only_frame(cfg, fset, *, cache)` helper would be a natural next step; deferred here because the `assemble()` contract is exactly one caller (the Stage 3 `weather_only` path via the CLI) and we did not want to touch it at a stage boundary. `build()` itself is feature-set-agnostic — it reads only `config.forward_fill_hours` and `config.name` — so both orchestrators share it without further refactor.
- **`_NamedLinearModel` wrapper in `train.py`, not a `LinearConfig.name` field.** The plan §6 Task T5 explicitly said "pick (b): keep `LinearModel` unchanged", so adding a `LinearConfig.name: str | None` field and threading it through `LinearModel.__init__` was off the table. The shipped approach is a thin wrapper class (`_NamedLinearModel`) in `train.py` that composes `LinearModel`, forwards `fit` / `predict` / `save`, and overrides the `metadata` property via `base.model_copy(update={"name": self._metadata_name})`. `load` raises `NotImplementedError` — the wrapper is stateless save for the name, so round-tripping it through the registry is not a meaningful operation and Stage 9 will own the registry-side story. `ModelMetadata` itself is `frozen=True` so `model_copy` is the idiom the Pydantic v2 docs point at. Tests pin the substitution end-to-end via `test_train_cli_features_override_swaps_feature_set`.
- **`features.weather_only` / `features.weather_calendar` as two optional slots, mutually exclusive at runtime.** An alternative shape would be a `FeatureSetConfig | None` on `AppConfig` directly, discriminated by a `name` tag, mirroring the Stage 4 `AppConfig.model` tagged union. We rejected that shape because `FeaturesGroup` already held a single `weather_only` slot (Stage 3), and a discriminated-union rewrite would touch every Stage 3 / Stage 4 consumer for no behavioural win. The two-slot approach keeps the Stage 3 surface byte-compatible; the mutual-exclusivity invariant is enforced once in `_resolve_feature_set`. Three tests in `test_train_cli.py` pin the four-way decision table (only-weather-only, only-weather-calendar, both, neither).
- **`conf/config.yaml` `defaults:` list flipped to group-swap form.** The Stage 3 default (`- features/weather_only@features.weather_only`) was package-override form: the YAML file was loaded and its contents placed at the package `features.weather_only`. For Stage 5 we need `features=weather_calendar` to swap *which file loads*, which is group-swap form (`- features: weather_only`). The refactor is one line in `conf/config.yaml` plus the optional-slot promotion on `FeaturesGroup`. Tests in `test_config.py` cover both default and override resolution; no Stage 3 / Stage 4 call-site needed changes because they only ever swapped values *inside* `weather_only.yaml`, not the group.
- **Holiday ingester persists every division; `derive_calendar` encodes only E&W and Scotland.** Policy-at-ingest vs policy-at-derivation: the parquet is what lands on disk for the next stage, and every caller can read it without re-ingesting. `derive_calendar` filters by division (`england-and-wales`, `scotland`) and passes `is_uk_wide = is_ew & is_sco & is_ni` (Northern Ireland intersection) into the `is_day_before_holiday` / `is_day_after_holiday` flags per plan D-4. Northern Ireland alone is not encoded because its ~3 % GB demand share is invisible in an aggregate-MW OLS; the plan's D-2 decision is preserved unchanged here. The design cost is that the persisted parquet carries ~30–50 rows of NI-only dates per year that no feature consumes — negligible on disk and policy-clean.
- **Pre-window WARNING on zero-filled rows, not a hard failure.** Plan D-6 mandates that pre-2019 training rows (before the gov.uk feed's coverage window) get zero-filled on the four holiday flags, not NaN and not raised. A single `loguru` WARNING per `derive_calendar` call names the affected row count; this matches the Stage 1 `UserWarning` pattern for unknown NESO columns. The feed's coverage window has narrowed since the original §R1 / §R10 research — the feed begins 2019-01-01 as of 2026-04, not 2012-01-02 — and zero-filling keeps the feature-table assembler usable on Stage 1's full 2018–2025 cached range without each stage having to re-research when gov.uk last re-opened the archive.
- **Calendar-module ownership of `CALENDAR_VARIABLE_COLUMNS`, re-exported through `assembler`.** `features.calendar` is the derivation module, so the ordered constant naming all 44 columns and their arrow types lives there. `assembler.py` imports and re-exports it so the 55-column `CALENDAR_OUTPUT_SCHEMA` has a single source of truth. Downstream `LinearConfig.feature_columns` reads the same re-export. No drift path: adding a 45th calendar column is one edit in `features/calendar.py`, which propagates to the assembler schema, the persisted parquet layout, and the linear regressor's regressor count automatically.

## Demo moment

From a clean clone (Stages 0-4 already built) with warm feature and holidays caches:

```
uv sync --group dev
uv run pytest                                                                                # all green
uv run python -m bristol_ml.ingestion.holidays --cache auto                                  # warms holidays cache
uv run python -m bristol_ml.train                                                            # weather_only (Stage 4 default)
uv run python -m bristol_ml.train features=weather_calendar                                  # Stage 5 group-swap
uv run jupyter nbconvert --to notebook --execute notebooks/05_calendar_features.ipynb --output /tmp/05_test_run.ipynb  # ~10 s
```

The two `train` invocations print per-fold metric tables whose headers name the active feature set; the notebook produces the 49-coefficient `results.summary()` block, the side-by-side metric means, the three-way benchmark (when the NESO forecast cache is warm), and the residual-ripple figure that motivates Stage 6.

## Deferred

- **Shared `_compose_weather_only_frame` helper.** Called out above. `assemble_calendar` duplicates the NESO/weather/resample/`build` sequence from `assemble()` inline. Factoring a shared helper is a cheap future refactor (both orchestrators are pure compositions); deferred to avoid touching the Stage 3 `assemble()` surface at a stage boundary.
- **Registry-side round-trip for `_NamedLinearModel`.** The wrapper's `load` raises `NotImplementedError`; round-tripping a named linear model through the Stage 9 registry is that stage's story. The Stage 4 `LinearModel.save` / `load` path is unaffected.
- ~~**`assembler` CLI dispatch by feature set.** The `python -m bristol_ml.features.assembler` CLI still only calls `assemble()`. Invoking `assemble_calendar` for Stage 5 cache warming is a one-line Python script (`assembler.assemble_calendar(cfg, cache='auto')`) or the `train` CLI's upstream cache check. Extending the assembler CLI to dispatch by `_resolve_feature_set` is a small follow-up not required by Stage 5 acceptance.~~ **Closed 2026-05-04** on branch `fix/feature-cache-regeneration-ux`. A new `_resolve_orchestrator` helper in `assembler.py` mirrors `train._resolve_feature_set`'s mutual-exclusivity rule, so `uv run python -m bristol_ml.features.assembler features=weather_calendar --cache offline` regenerates the Stage 5 cache directly. The fix landed in response to a real failure mode: a contributor adding a calendar column (`is_dst`, commit `e448e81`) hit `Cached parquet ... is missing required column 'is_dst'` and had no actionable CLI to recover with — the pre-fix train-CLI error message named `assemble_calendar` (a Python function, not a CLI verb), which was directly misleading. The fix also augments the schema-mismatch errors in `load` / `load_calendar` to include the regen command, adds a "Regenerating feature caches after a code change" section to `README.md`, and extends the module guide.
- **`is_school_holiday_ew` / `is_school_holiday_sco` flags.** External research §R4 flagged school-term boundaries as a secondary calendar effect (half-term weeks, summer holiday); Stage 5 intentionally defers these because the gov.uk feed does not carry them and the alternative sources are per-local-authority, fragmented, and not under an open licence. A future Stage 6/8 enrichment could source from `dfe.gov.uk` term-date data; out of scope here.
- **Feature interactions and lag features.** A main-effects-only OLS cannot see "hot Saturdays vs cold Mondays" (interaction) or "yesterday's demand predicts today's" (lag). Stage 6 lifts the linearity constraint with tree-based models; Stage 7 adds explicit lag features. The Stage 5 notebook's closing narrative names both as teasing for the next stages.
