# Stage 3 — Feature assembler + rolling-origin split

## Goal

Land the first end-to-end feature table — demand + population-weighted national weather, hourly, UTC — and the rolling-origin enumerator that every subsequent modelling stage (Stages 4+) will iterate over. Introduce the features layer's canonical `OUTPUT_SCHEMA` mechanism and the evaluation layer as a new module peer.

## What was built

- `src/bristol_ml/features/assembler.py` — the module. Public surface: `OUTPUT_SCHEMA: pa.Schema`, `build(demand_hourly, weather_national, config, *, neso_retrieved_at_utc=None, weather_retrieved_at_utc=None) -> pd.DataFrame`, `load(path) -> pd.DataFrame`, `assemble(cfg, cache="offline") -> Path`, and `_resample_demand_hourly(df, agg) -> pd.DataFrame`. Writes atomic parquet to `${BRISTOL_ML_CACHE_DIR:-data/features}/<feature_set>.parquet` via `ingestion._common._atomic_write`. Emits one structured loguru INFO line per `build()` call covering `feature_set`, `row_count`, `demand_nan_rows_dropped`, `weather_forward_filled_rows` (per variable), `weather_nan_rows_dropped_after_fill` (per **D5**). Module CLI `python -m bristol_ml.features.assembler [--cache auto|refresh|offline]`.
- `src/bristol_ml/evaluation/__init__.py` + `src/bristol_ml/evaluation/splitter.py` — the evaluation layer's first module. Public surface: `rolling_origin_split(n_rows, *, min_train, test_len, step, gap=0, fixed_window=False) -> Iterator[tuple[np.ndarray, np.ndarray]]` and `rolling_origin_split_from_config(n_rows, SplitterConfig)`. Expanding-window default per **D4**, with the `fixed_window` knob for a sliding variant. Module CLI `python -m bristol_ml.evaluation.splitter [--n-rows N]`.
- `conf/_schemas.py` — four new Pydantic models (`FeatureSetConfig`, `FeaturesGroup`, `SplitterConfig`, `EvaluationGroup`) and two `AppConfig` fields (`features: FeaturesGroup`, `evaluation: EvaluationGroup`). `FeatureSetConfig.demand_aggregation: Literal["mean", "max"]` per **D1**; `SplitterConfig.fixed_window: bool = False` per **D4**. Both follow the `IngestionGroup | None = None` sibling pattern.
- `conf/features/weather_only.yaml` — Hydra group with `# @package features.weather_only` header; `name: weather_only`, `demand_aggregation: mean`, `cache_dir: ${oc.env:BRISTOL_ML_CACHE_DIR,data/features}`, `cache_filename: weather_only.parquet`, `forward_fill_hours: 3`. - `src/bristol_ml/features/__init__.py` — docstring updated to note that the assembler module is accessed by submodule import (`from bristol_ml.features import assembler`) rather than a top-level alias, matching the ingestion-layer convention for modules with a wide public surface.

## Design choices made here

- **Loguru for the D5 structured log, not `logging.getLogger(__name__)`.** The plan §6 Task T3 text named `logging.getLogger(__name__)` verbatim, but every shipped module in the repo (`ingestion.neso`, `ingestion.weather`, `features.weather`) uses loguru. Matching house style is the lower-surprise choice; the plan's intent — "one structured line at INFO on every `build()` call, greppable in retros" — is satisfied identically. The test cost is a small caplog-adapter fixture (a standard loguru idiom, seven lines of `conftest.py`) rather than a zero-lines `caplog` dependency. Surfaced at T3 completion; accepted.
- **Programmatic test fixtures instead of hand-rolled parquet.** Plan §7 "New" listed `tests/fixtures/features/toy_feature_inputs/demand.parquet` as a hand-crafted asset. We generate the test inputs in-fixture from `neso.OUTPUT_SCHEMA` and `weather.OUTPUT_SCHEMA` instead, which matches the Stage 2 risk-register mitigation verbatim ("Generate fixture from `neso.OUTPUT_SCHEMA` programmatically in a `conftest.py` fixture, not hand-rolled"). Fixture drift becomes impossible because the on-disk schemas are the source of truth; spring-forward and autumn-fallback fixtures are parameterised date ranges rather than separate binary files.
- **`assemble()` consumes a Pydantic `AppConfig`, not a Hydra `DictConfig`.** The orchestrator signature is `assemble(cfg: AppConfig, cache: str = "offline") -> Path`. The CLI wrapper is the only place Hydra composition runs; tests build `AppConfig` instances directly. This was forced by the T4 first attempt, where a subprocess-invoked CLI test tried to override `ingestion.weather.end_date` and hit `ConfigCompositionException` because the field has no YAML default (it is `date | None` with `None` default). Bypassing Hydra in the test keeps the brittle override syntax out of the test surface entirely; the subprocess `--help` smoke is retained separately to exercise CLI wiring.
- **Notebook Hydra-override for the warm-cache size mismatch.** The shipped default `evaluation/rolling_origin.yaml` picks `min_train_periods=8760` — one full year of hourly data, assuming multi-year training windows common in production backtests. The Stage 2 weather warm cache ships one calendar year of data (2023, 8,760 hours), and an inner join with the multi-year NESO cache yields exactly 8,760 rows. The splitter correctly refuses the resulting infeasible config (`min_train=8760 + test_len=24 > n_rows=8760`). The notebook resolves the mismatch with an in-line Hydra override `evaluation.rolling_origin.min_train_periods=720` (30 days) and a markdown paragraph explaining the override and why production would drop it. The alternative — carving a Stage-3-specific `rolling_origin.yaml` variant — would have been an config-tree bulge serving only the demo surface.
- **`cloud_cover` widens to `float32` in the national aggregate, not `int8`.** Source `weather.OUTPUT_SCHEMA` stores `cloud_cover` as `int8` (0-100 %). `national_aggregate` takes a weighted mean, which may produce fractional values, so the assembler's `WEATHER_VARIABLE_COLUMNS` declares `cloud_cover` as `float32`. Documented in `features/CLAUDE.md`; the alternative (rounding back to int8 after aggregation) loses information for no storage win.
- **Two provenance columns, scalar per run.** `neso_retrieved_at_utc` and `weather_retrieved_at_utc` are populated once per `build()` call (max of the respective source frame's `retrieved_at_utc` column, or overridden via kwarg for CI determinism). Cheaper than a metadata sidecar; honours **D8** and DESIGN §2.1.6; and means `test_assembler_deterministic` can assert byte-equality modulo these two columns via kwarg.
- **Expanding window default with `fixed_window` config knob.** `rolling_origin_split` yields `(train_idx, test_idx)` with `len(train)` growing by `step` per fold under the default. When `fixed_window=True`, the training window slides forward keeping `len(train) == min_train`. `sklearn.TimeSeriesSplit` semantics familiar to newcomers; matches the DESIGN §5.3 "mean-and-spread across folds" language naturally. No sklearn dependency added.
- **`rolling_origin_split_from_config(n_rows, config)` helper.** The splitter proper takes primitive keyword arguments, not a `SplitterConfig`, so tests and downstream code can call it without constructing a Pydantic model each time. The `_from_config` wrapper is the sugar the notebook and CLI reach for. Mirrors `rolling_origin_split` proper keeping its parameters symmetric with what a scikit-learn user expects.
- **Atomic parquet writes via `ingestion._common._atomic_write`.** Reuses the Stage 2 helper rather than adding a features-layer equivalent. Satisfies AC-1 under repeated `assemble()` calls; no `.tmp` leaks observable on the filesystem.

## Demo moment

From a clean clone (Stages 0-2 already built):

```
uv sync --group dev
uv run pytest                                                               # 183 passed
uv run python -m bristol_ml.features.assembler --help                       # CLI help
uv run python -m bristol_ml.evaluation.splitter --help                      # CLI help
uv run python -m bristol_ml.features.assembler --cache auto                 # first assemble (~1 s on warm caches)
uv run jupyter nbconvert --to notebook --execute notebooks/03_feature_assembler.ipynb --output /tmp/03_test_run.ipynb  # ~6 s
```

The notebook produces two plots — a year-long view of GB National Demand with rolling-origin fold boundaries overlaid, and a week-long zoom showing seven consecutive 24-hour test folds.

## Deferred

- **`weather_calendar` feature set.** The second feature set (Stage 5) will add calendar features alongside `weather_only`; the assembler will grow a calendar join step. The `features.weather_calendar` Hydra group slot is already reserved in the D3 decision record but no config file ships at Stage 3.
- **Metric functions (`mae`, `mape`, `rmse`, `wape`).** Belong to the evaluation layer but are Stage 4 scope. `evaluation/__init__.py` is explicitly minimal so a Stage 4 `evaluation/metrics.py` drops in without conflicting with the splitter.
- **NESO benchmark comparison.** Where the day-ahead NESO forecast feeds into evaluation — as a `benchmarks` module peer or as an evaluation-layer sibling — is deferred to Stage 6; flagged on `docs/architecture/layers/evaluation.md` as an open question.
- **Multi-horizon splits.** The splitter yields fixed `test_len`-hour horizons; multi-horizon fan-out (day-ahead + week-ahead on the same fold structure) lands when a model that needs it does.
- **The dead `columns` field on `NesoIngestionConfig`.** Carried forward from Stages 1 and 2; no Stage 3 cause to touch it.
- **Known drift, surfaced for the lead (do not fix from within Stage 3):**
  - DESIGN §9 implies the features layer arrives at Stage 3; Stage 2 shipped its first member early (the weather aggregator). The Stage 3 additions are the first doubling of the layer. Noted historically in the Stage 2 retro; no §9 edit required since the shipped ordering is compatible with the spec's Stage 4 consumer.
  - DESIGN §6 layout tree still missing Stages 1 + 2 additions; Stage 3 adds `features/assembler.py`, `evaluation/__init__.py`, `evaluation/splitter.py`, `conf/features/`, `conf/evaluation/`. Deny-tier for the lead; batched human edit.
  - `docs/architecture/ROADMAP.md` "Features" section's open questions are answered in Stage 3 (feature-table schema contract ≈ `OUTPUT_SCHEMA`; population-weighting home ≈ features layer). Deferred to a ROADMAP pass at Stage 5/6; non-blocking.
